Примерно так. Сколько по факту producer, consumer может быть задано в конфигах приложений. И быть не по 1 consumer на сервис, а по 10, например

Если замеряется производительность только асинхронной части, которая берет задачи из очереди и кладет ответы в очередь, то ее производительность удобнее замерять по этим timestamp или по логам/метрикам этой асинхронной части

Если эта часть системы сама по себе работает без других очередей, а просто берет задачу, обрабатывает, сохраняет ответ. То грубо ее можно тестировать по consumer lag: загрузить в очередь 1 000 000 задач, сдвинуть метку чтение для consumer group на 0 и запустить систему. И подождать, когда по метрикам Prometheus например, consumer lag для очереди задач уменьшится с 1 000 000 до 0. Пусть это произойдет примерно за 50 минут, значит скорость обработки - 1 млн за 50 минут или 3 мсек на задачу в среднем, что просто огонь, например

Потом поставить новую версию систему, снова сдвинуть указатель и запустить обработку задач снова с 1 000 000 до 0, пусть стало 300 мсек на задачу - деградация х100

ℹ️ k6 позволяет описывать тест как код, поэтому большое значение имеют примеры кода

Пример проекта на k6, который пошагово демонстрирует возможности инструмента:
https://github.com/SwissLife-OSS/K6-MultiScenario-template/tree/main/learnk6

Проект обучающий, начального уровня. Аналогичные примеры есть и в документации. Документация на k6 обширная и нужно привыкнуть к навагации по ней - нужный пример может находиться в 4-х местах

▫️ В разделе Guides / Using k6
▫️ В разделе примеры https://k6.io/docs/examples/
▫️ В описании протокола или класса (редко, но есть), например
- k6 API / k6/net/grpc
- k6 API / Counter
- k6 API / http / Response
- k6 API / experimental / websocket
▫️ В описании конкретного метода класса (почти всегда есть):
- k6 API / http / asyncRequest
- k6 API / http / request
- k6 API / http / get
- k6 API / http / post
- ...

И этой документации, достаточно, чтобы написать тесты. Кодом

ℹ️ k6 позволяет многие аспекты нагрузки передавать через аргументы, то есть не задавать их в коде вообще, no code, и этим рекомендую пользоваться

Описание options находится в главном разделе Guides / Using k6 и демонстрирует несколько вариантов задания для:
- RPS (интенсивность нагрузки)
- Iterations (длительность теста, количество выполнений функции default)
- VU (количество подключений, удобно считать VU = RPS х 5 или RPS х10)
и других параметров

Это наиболее часто изменяемые параметры для простого теста стабильности, и их можно не фиксировать в коде, а использовать один из вариантов передачи параметров в тест

ThreadGroup

If controller (для покупки билета туда-обратно в половине случаев)
