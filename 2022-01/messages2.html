Более сложная работа с токенами.

Например, мы знаем что jwt токен выдается на 600 сек. А шаг нагрузки транзакции 200 сек.
Значит, каждый токен можно будет использовать, если на момент начала его использования у него останется еще 200 сек жизни.

Простой формат синхронизации потоков по работе с токенами такой.
В JSR 223 при получении токена переменную со значением
def stop = System.currentTimeMillis() + 600_000 - 200_000;
Положить в очередь структур
def tokenMap = [:]
tokenMap["stopTime"] = stop
tokenMap["token"] = jwtToken
tokenMap["username"] = userName
props.get("tokenQueue").put(tokenMap)


А потом в другом JSR-223 при получении токена с целью использования, можно делать вот так

def tokenMap = props.get("tokenQueue").take()
while(tokenMap["stopTime"] > System.currentTimeMillis()) {
    tokenMap = props.get("tokenQueue").take()
}
props.get("tokenQueue").put(tokenMap);
vars.put("token", tokenMap["token"])
vars.put("username", tokenMap["username"])

Сбор статистики на высоких RPS
Не исследовал самостоятельно как можно удобно собирать статистику в Locust. То что видел - отправка статистики по каждому запросу. Это будет удобно только если интенсивность невысокая - 1-50 TPS (это как Raw Backend Listener в JMeter), если интенсивность выше 100 TPS, то использовать такой подход не стал бы. За секунду будет отправляться несколько сотен запросов на вставку в TimeSeries хранилище, которое может легко стать узким местом процесса НТ. В описании говорится, что протестирована отправка 5000 RPS - это надо проверять:
https://github.com/SvenskaSpel/locust-plugins/tree/master/locust_plugins/timescale
Вот так реализован мониторинг, на каждый запрос будет отправлен INSERT в PostgreSQL:
https://github.com/SvenskaSpel/locust-plugins/blob/2.4.0/locust_plugins/listeners.py#L169
Резюмируя - для невысокой нагрузки инструмент использовать можно, как есть. А для высокой, сначала бы в файл просто писал, а после теста отправлял бы куда-нибудь.

Баги
Могут быть дефекты в инструменте. Например вот этому коду не доверяю
https://github.com/SvenskaSpel/locust-plugins/blob/2.4.0/locust_plugins/listeners.py#L156-L157
                samples_buffer = self._samples
                self._samples = []
При записи результатов без блокировки сначала буфер результатов копируется в локальную переменную samples_buffer и в следующей строке очищается. Между этими двумя строками при высокой интенсивности в samples добавится уже множество результатов, которые будут очищены во второй струке и просто пропадут.
Ведь, при этом, на каждый запрос выполняется такой код:
https://github.com/SvenskaSpel/locust-plugins/blob/2.4.0/locust_plugins/listeners.py#L235
        self._samples.append(sample)
Эти два участка кода (запись результатов с очисткой и добавление) конкурируют за один и тот же общий объект, который является простым списком
https://github.com/SvenskaSpel/locust-plugins/blob/2.4.0/locust_plugins/listeners.py#L64:
        self._samples: List[dict] = []
Это потенциальная бага. Такого может быть много где. Важнее, чтобы такого в самих сервисах не было, если команда такое в сервисах не допускает, то и этот момент сможете исправить в инструменте. Плюс тут в том, что этот код можно поправить, он открыт
