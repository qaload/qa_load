Заметил проблему в Throughput Shaping Timer 2.5 для JMeter для 0-х rps

Если написать профиль такой, чтобы был рост нагрузки с 0 до rps а потом с rps до 0, то в конце теста у таймера задача сделать нагрузку для 1-2 потоков на rps = 0.0

Пауза в миллисекундах вычисляется вот так, где cntDelayed = 1-2, rps = 0.0 :
return (int) (1 + 1000.0 * (cntDelayed + 1) / rps);

И так получается, что при делении на вещественный 0 не кидается исключение, а получается значение = MAX_INT.

Ранее делал доклад и проект по визуализации статистики по запросам к PostgreSQL. Ссылки в сообщении, на которое ссылаюсь.

Алгоритм такой:
1. Раз в 10 минут делается запрос к pg_stat_statements, сохраняется в timeseries db (influxdb)
2. Далее из собранной статистики вычисляется дельта, в отдельную таблицу сохраняется только те запросы, по которым были изменения (которые выполнялись за последние 10 минут)

Есть с этим решением проблема - оно может начать тормозить примерно через год после начала использования.
Причина оказалась в данных на шаге 1 (я так думаю сейчас, это не точно).

Если периодически не выполнять процедуру
pg_stat_statements_reset()
но целый год ставить новые версии сервисов и менять код, то в статистику сервера БД попадут несколько версий одного и того же запроса

select id as "_int_field_1" from table where _int_field_1 = ?
select id as "_int_field_2" from table where _int_field_2 = ?
select id as "_int_field_3" from table where _int_field_3 = ?

Например, потому что ORM-фреймворк так в этой версии решил его сгенерировать - чуть иначе

И их может стать 10 000 например в pg_stat_statements, по 500 из них статистика будет расти, а по 9500 - нет

Что можно сделать - при настройке pg_stat_statements указывать размер истории не более 500-1000.
ALTER SYSTEM SET track_activity_query_size = 1000;
Или указывать большое число, но например, каждую ночь выполнять 
select pg_stat_statements_reset();

Переводя проблему на технические термины InfluxDB - если кардинальность нескольких ключей InfluxDB начинает превышать 5000, а вы постоянно делаете запросы и группировки по этим ключам, то InfluxDB начинает потреблять всю доступную ОЗУ на сервере. Чтобы держать кардинальность на уровне 500-1000 можно не допускать попадания в теги ненужных значений
