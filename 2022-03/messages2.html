Привет!
По ошибке "code: java.net.SocketException"

httpclient.reset_state_on_thread_group_iteration=false
https://github.com/apache/jmeter/blob/v5.5-rc1/bin/jmeter.properties#L104

Вот эта опция должна помочь. Ее надо рассментировать и поставить в false.
Думаю остается вопрос - почему она должна помочь?

Если на нагрузочном агенте не переиспользуются подключения к серверу, то они закрываются и открываются снова. Но закрытие происходит долго. Дольше открытия иногда. И поэтому они заканчиваются

Чтобы было все ок с подключениями
Включите Use KeepAlive (не отключайте) в
https://jmeter.apache.org/usermanual/component_reference.html#HTTP_Request

Чтобы хватало подключений
Включите Same user on each iteration (не отключайте) в
https://jmeter.apache.org/usermanual/component_reference.html#Thread_Group

Если не используется Thread_Group а используется что-то другое, то тогда аналог:
httpclient.reset_state_on_thread_group_iteration=false
в конфиге:
https://github.com/apache/jmeter/blob/v5.5-rc1/bin/jmeter.properties#L104

В отправляемом сообщении не стоит заголовок JMSReplyTo
Получается вы знаете в какую очередь отправляете, знаете из какой очереди будете читать ответ, но не сказали серверу в какую очередь отправлять ответ.

Иногда это не нужно делать - если сервер - это такой сервис, который всегда из одной очереди читает и в одну пишет, и клиент у него тоже один, а заголовок запроса JMSReplyTo просто игнорируется. Или клиентов много, но они читают ответы из общей очереди по СorrelationID, по другим признакам выбирают свои ответы

Но бывает, что это не так. Что очередь для приема сообщений - одна на весь сервис, а вот очереди с ответами создаются для каждого клиента сервиса свои, чтобы клиенты получали только свои ответы. И CorrelationID используется только чтобы сопоставить свой запрос со соответствующим ответом

Вот эта уникальная очередь своих ответов задается самим клиентом. В заголовке запроса

В общем - заполните заголовок JMSReplyTo значением MQ_LS_NAME

И в документации сказано следующее:
https://gatling.io/docs/gatling/reference/current/jms/#other-options
  // optional, default to 1
  // listener thread count
  // some JMS implementation (like IBM MQ) need more than one MessageListener
  // to achieve full readout performance

  // optional, default to `matchByMessageId`

  // Use `matchByCorrelationId` for ActiveMQ.

Для IBM.MQ нужно задать больше чем 1 поток. Сколько именно не знаю, пусть 3. Но самое важное тут для получения сообщений - по умолчанию в методе requestReply будет ожидаться ответ, по которому будет совпадение по matchByMessageId если у вас не ActiveMQ.

И получается, если в протоколе не задать явно
.matchByCorrelationId()
а при этом корреляция выполняется именно по CorrelationId, и получится так, что придет ответ, в котором MessageID ответа будет отличаться от MessageID запроса. То метод requestReply будет вечно ждать ответа с нужным MessageID.

  // optional, default to 1
  .listenerThreadCount(1)

  .matchByCorrelationId()
