Vyacheslav Smirnov
Если за -c принять количество одновременно работающих пользователей.

То время Time per request (mean) - среднее время ответа в контексте одного такого пользователя/потока. Прошло 100 секунд теста, один поток получил 100 ответов, среднее время 1 сек. Такая метрика есть только в ab.

Если бы кто-то был именно этим пользователем. И отправлял из своего браузера запросы по одному. То он бы получал ответ на каждый запрос через 1 сек.

Time per request (mean, across all concurrent requests) - среднее время между ответами, которые получает инструмент ab. Такая метрика есть только в ab.

Прошло 100 секунд, работало 700 потоков, каждый получил по 100 ответов, всего ответов было получено 70 000, среднее время между получениями ответов - 100 / 70 000 = 1,428571 мсек

Это некая синтетическая метрика. Которая отражает производительность сервера.

Если производительность сервера постоянная и провести три теста

ab -n 70000 -c 10 http://localhost/

ab -n 70000 -c 100 http://localhost/

ab -n 70000 -c 700 http://localhost/

То Time per request (mean, across all concurrent requests) во всех трёх тестах будет постоянной. Серверу нужно было обработать 70 000 запросов, он будет посылать ответы каждые 1,428571 сек.

А время Time per request (mean) будет расти с ростом параллельности отправки запросов. Чем больше потоков, тем медленнее каждый из них будет видеть свой ответ.

Поправил описание. Сначала неверно написал. Потом сделал такой же тест в JMeter, сравнил показания. JMeter покажет в своем отчёте Time per request (mean) в среднем времени отклика.

Значит интенсивность теста в сценариях будет такой: TPS = RPS / RequestCount = 1000 / 15 = 67 сценариев в сек в пике.

Если предположить, что максимальное время выполнения запроса под нагрузкой 1000 RPS будет 10 сек (некий таймаут). А после такого времени балансировщик начнёт отвечать 503 Сервер недоступен (обычно этот таймаут больше, 10 сек, но это детали). То будем считать MaxTimeRequest = 10 сек.

Тогда MaxTimeScenario = Request * MaxTimeRequest = 150 cек.

Чтобы был некий запас, время на работу самого Apache.JMeter, а не только время на ожидание ответов и отправку запросов добавим немного. Получится, что Pacing = MaxTimeScenario + немного = 200 сек.

Дано:
TPS = 67 сценариев в сек
Pacing = 200 сек.

Необходимое количество потоков:
Threads = TPS / (1 / Pacing) = TPS * Pacing = 67 * 200 = 13400 в пике.

Если запросов в сценарии больше. И тайминги выше, то потоков понадобится ещё больше. Если тайминги ниже, и ответы будет существенно быстрее, чем через 10 сек, то потоков понадобится меньше.

Оптимизация! Обратите внимание на вариант, где Loop Count был равен 10, а не 1. В этом варианте один поток создаётся чтобы выполнить 10 сценариев в цикле. Этот вариант в 10 раз экономичнее чем вариант с Loop Count = 1. Так как ресурсов тратится значительно меньше. Его минус такой. При Loop Count = 1 мы точно знаем, что расписание запуска будет стабильным. Каждую сек один поток создаётся, делает одну итерацию и закрывается. RPS будет более-менее ровный. При Loop Count = 10 поток будет выполнять 10 итераций. По условию задачи, таймеры мы не используем. Потому нагрузка будет менее ровной, но средний RPS на большом периоде времени (10 минут) будет такой же гладкий.

Также обратите внимание на
[✓] Delay thread Creation Until needed
Это позволяет создавать потоки только в нужный момент. К моменту их запуска, предыдущие с высокой вероятностью уже отработают и закроются. И общее количество активных потоков будет невысоким в каждый момент времени.

Чтобы как-то компенсировать зависание можно выставить Duration Assertion на ответы. Или на транзакцию. И в настройках Thread Group настроить - остановку итерации или потока при ошибке.

Duration Assertion на Transaction Controller будет работать только если выставить галочку Generate Parent Sampler. Это не самое оптимальное действие. При такой настройке JMeter будет копить в памяти статистику по запросам внутри транзакции, до ее завершения. И при большом количестве запросов памяти может не хватить. Но при небольших нагрузках - рабочий вариант.

Тогда при срабатывании Duration Assertion произойдет ошибка, Thread Group выполнит действие указанное в настройках: Action to be taken after a Sample Error, и итерация остановится, нагрузка спадёт. Потоки не накопятся. JMeter не упадёт.
