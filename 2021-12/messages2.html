Это самое простое и безотказное решение когда у нас предсказуемое время выполнения итераций.

Второй способ - тоже самое но вместо countra мы работаем с временем:
1) задаем переменной expiredtime=0 для начала
2) делаем IF (стандартный элемент) и проверяем что currenttime >= expired time
3) если 2) истинно - то генерим новый токен и задаем новое значение в expiredtime=(currenttime+1hour-coupleminutes)
Таким образом IF будет выполнен в самом начале чтобы взять первый токен и далее будет периодически выполнятся каждый час (минус пару минут чтобы успеть до того как просрочится)

» Подскажите плз, в постгресе есть explain analyze - выводит реальный план запроса

explain выводит один из возможных планов запроса, да реальных, запрос выполняется. Если строите план update, insert, delete-запроса, то поместите запрос в транзакцию, для которой после выполнения сделайте rollback.

Вот почему один из возможных - при выполнении запроса под нагрузкой у PostgreSQL может быть одна информация по статистике и одни веса, он строит план1.
После теста статистика может быть пересчитана и параметры у запроса (вы же подставляете конкреные параметры) могут отлиаться от параметров на бою или под нагрузкой - и будет другой план. План 2

Есть вероятность, что план 2 совпадет с планом 1

Чтобы получать план в момент выполнения и видеть план1, можно использовать расширение auto_explain
https://postgrespro.ru/docs/postgresql/9.6/auto-explain

» если запрос будет ресурсоёмкий, то не повредит ли текущей производительности сервера
Да, может. Если сделать запрос вида
select id from (select * from table1 cross join table2) то может например переполниться tempdb, в которую будет помещен результат select * from table1 cross join table2 и тогда замедлится или откажется работать весь сервер, до момента завершения запроса.
Также запрос может загрузить 1 ядро или несколько ядер CPU (при активации parallel scan) в ходе выполнения.
И заберет одно подключение к БД.
И также может нагрузить диски
Также может поставить блокировки на некоторые записи в таблицах - зависит от запроса

Есть решения по эмуляции запросов на копии продуктива.
Например такое: https://t.me/databaselabru
Тут идея в том, что разработчик отправляет боту запрос, бот выполняет запрос на копии продуктива, но в ответ разработчику отдает не результат запроса, а только план. Нет раскрытия продуктивных данных в явном виде. Можно получить информацию неявно по принципу есть/нет, например по запросу explain select * from accounts where userName='Смирнов В.А.' and accountBalance > 1000000 можно узнать больше миллиона на счету пользователя или нет. Но защищаться от разработчика сложно, нужно доверять.

Решение с auto_explain оптимальное. В нем будет раскрытие параметров запроса, но не будет результатов. И повторного запроса нет - сразу план запроса. Тоже не идеальное, просто оптимальное

Если часто делаются выборки из хранимой процедуры, которая тоже делает select (просто скрытый внутри), то перекомпиляция будет нагружать CPU просто так.
Кеш плана запроса не будет использоваться ведь

Если в таблицу делается много вставок или делаются update/delete запросы, а потом из нее выбирает данные хранимая процедура, то тогда перекомпиляция может помочь.
Но, план запроса при перекомпиляции будет опираться на статистику по таблицам. И эта статистика тоже должна быть актуальной, иначе при перекомпиляции (обновлении плана) будет план на основе старой статистики. И тогда перекомпиляция даст прежний результат.

Обновлять статистику по таблицам можно ночами.
https://docs.microsoft.com/ru-ru/sql/relational-databases/system-stored-procedures/sp-updatestats-transact-sql?view=sql-server-ver15
явным вызовом соответствующего API

Но это детали

А в общем, да, перекомпиляция может не приводить к ускорению.
- если в исходных данных не было изменений
- если в исходных данных изменения были, но статистика не обновилась
- если очень мало ресурсов CPU на сервере из-за неоптимальных запросов (других) и из-за них же может медленнее работать текущая SP, не из-за перекомпиляции, а по другим причинам
