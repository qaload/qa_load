а в сам тест план необходимо добавить autoStopLogic() в корень, он вернет постпроцессор который будет выполняться после каждого семплера. из котлин файла надо выпилить вызовы QualityGate.result() - это моя интеграция локальная.
описательная часть :
Метод AutoStop.autoStopOnCondition принимает на вход полную дичь, но в этом не сложно разобраться.

Аргументы по порядку:

DslHttpSampler (расширим до любого элемента) - это тот семплер который вы описываете в классе Samplers или любой DslHttpSampler, который вы можете написать хоть рядом в этом же блоке, от него возьмется его название в виде стринги. Альтернатива - просто передавать стрингу, как это показано на примере выше. Также, у метрики есть зарезервированное слово - “all“, в этот лейбл пишутся метрики всех запросов внутри вашего теста.

Тут чуть сложнее. В этот блок передается метрика. Все метрики передаются через конструкцию общий класс метрик + квадраточие (мне тоже нравится это название) + имя метрики. StatsSummary::sampleTime - вот пример метрики, которая содержит время отклика запросов возвращает. Какие еще есть метрики? Их много, а еще их можно расширить заэкстендив класс StatsSummary. Вот список:

receivedBytes - получено байт, вернет агрегат CountMetricSummary

sentBytes - отправлено байт, вернет агрегат CountMetricSummary

sampleTime - общее время отклика запроса, вернет агрегат TimeMetricSummary

connectTime - время создания коннекта, вернет EmbeddedTimeMetricSummary

errorPercent - процент ошибок, вернет DetailedEmbeddedTimeMetricSummary

idleTime - время ожидания, вернет EmbeddedTimeMetricSummary

latencyTime - в нашем инструменте это TTFB, вернет EmbeddedTimeMetricSummary

responseCode - код ответа, вернет DetailedEmbeddedTimeMetricSummary

errors - только ошибочные запросы, вернет CountMetricSummary

samples - все запросы, вернет CountMetricSummary

Самое сложное это третий аргумент, он говорит о том, какую сделать агрегацию над метрикой. Для этого выше в списке я писал какие возвращаются агрегаты и каждый из них умеет делать свой набор агрегаций. Выглядит аналогично второму класс агрегата + квадроточие + метод агрегации EmbeddedTimeMetricSummary::perc95

CountMetricSummary

total - общая сумма, возвращает Long

perSecond - событий в секунду, возвращает Double

TimeMetricSummary - всегда возвращает Duration

max - максимальное значение

mean - среднее значение

min - минимальное значение

median - 50-ый перцентиль

perc90 - 90-ый перцентиль

perc95 - 95-ый перцентиль

perc99 - 99-ый перцентиль

EmbeddedTimeMetricSummary - тоже самое что и TimeMetricSummary

DetailedEmbeddedTimeMetricSummary

errorPercent - необходимая агрегация для получения метрики errorPercent (так уж совпало), вернет Double

instantValue - необходимая агрегация для получения метрики responseCode, вернет String

Самое главное это финал. В конце мы передаем условие остановки. Оно представлено великим разнообразием проверок библиотеки org.hamcrest.Matchers, который можно в каком то виде увидеть тут. Примеры могут быть такими:

greaterThan(2L) - останавливаем, когда значение выше двух

lessThan(2.seconds) - останавливаем, когда значение ниже 2х секунд

equalTo("404") - останавливаем, когда значение равно 404

В итоге передав 4 аргумента, мы говорим, что мы хотим вот с этих запросов (1) собрать вот такую метрику (2), вот так её проагрегировать (3) и остановить тест если она будет не соответствовать вот этому правилу (4)

            label = result.sampleLabel
            responseCode.instantValue = result.responseCode
            connectTime.add(result.connectTime)
            latencyTime.add(result.latency)

                last = value
            }
            fun last() : Double {
                return last
            }
