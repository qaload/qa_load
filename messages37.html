Since 3.0, Gatling requires Scala 2.12. Gatling is not compatible with Scala 2.11 nor Scala 2.13.

enablePlugins(GatlingPlugin)
scalaVersion := "2.13.4"
scalacOptions := Seq(
  "-encoding", "UTF-8", "-target:jvm-1.8",
 "-deprecation", "-feature", "-unchecked",
  "-language:implicitConversions", "-language:postfixOps")
val gatlingVersion = "3.5.0"
...
Удобно смотреть версию scala по файлу build.sbt проекта gatling-sbt-plugin-demo для нужной версии Gatling.

Об этом поле есть статья в блоге blazemeter.

Я использовал поле X-Forward-For, которое заполняется через Header Manager. Это позволяет использовать неограниченное количество ip-адресов, притворяться адресами из разных регионов, самый простой для меня способ.
Описание:
https://t.me/qa_load/10090

Суть в том, что для поддержки работы клиентов через прокси-серверы у HTTP-серверов есть поддержка такого поля. Прокси-сервер передает в поле оригинальный IP-адрес клиента, а запрос приходит серверу с ip-адреса прокси-сервера. Все это на уровне HTTP-протокола.

И если в настройках сервера приложений указать, что адрес балансировщика (HaProxy/NGinx/...) - это доверенныq адрес, с которого можно принимать такие заголовки и доверять им. То JMeter/Gatling/LoadRunner может отправлять такие заголовки в HaProxyNGinx, балансировщик их примет и отправит серверу приложений, а тот если доверяет балансировщику их примет и будет считать адресами клиентов. А ответ будет возвращаться от сервера приложений к балансировщику, от балансировщика к JMeter/Gatling/LoadRunner. Сервер отправляет ответы, а балансировщик сам разбирается как их доставить клиентам, а клиентов на самом деле тут нет - нагрузочные станции их выдумали.

Поэтому если нужны тысячи ip-адресов, используйте этот способ

🔹Вопрос про конкурентную запись значений в Property в Apache.JMeter.
Вопрос решился при корректном использовании кавычек, и применении функции ${__P(prop_name)} взамен кода props.get("prop_name").
Вообще Property - это HashMap, это не ThreadSafe-структура, и не предназначена для многопоточной записи/обновления/удаления значений. Но читать значения в несколько потоков можно. Для многопоточной записи лучше использовать ConcurrentHashMap, которую можно положить в Property в начале теста, в Setup Thread Group, а потом можно много раз считывать и уже с таким объектом работать многопоточно.
Пример:
//Setup Thread Group: Положить Map в Property
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.LinkedBlockingQueue;
def map = new ConcurrentHashMap<String, String>();
props.put("map", map)
//Thread Group: Обновить значение
import java.util.concurrent.ConcurrentHashMap;
ConcurrentHashMap map = props.get('map');
def auth_token = vars.get('auth_token');
map.put('token', auth_token);
//Thread Group: Получить значение и положить его в переменную
import java.util.concurrent.ConcurrentHashMap;
ConcurrentHashMap map = props.get('map');
def auth_token = map.get('token');
vars.put('auth_token', auth_token);
