При большой нагрузке данные могут не записаться

одна из причин - очень большие пакеты данных, а в InfluxDB есть ограничение на размер пакета

Что можно сделать чтобы пакеты стали меньше?

- сделать меньше данных
- отправлять данные чаще

Как можно сделать данных меньше? Если в InfluxDB отправляются не сырые данные, а статистика, то размер статистики зависит от количества уникальных имен объектов, по которым строится статистика

В инструментах есть режимы - summary-only, есть в Gatling и JMeter, не будет имен запросов, будет только общая статистика, уже хорошо для online-анализа

Как начать записывать JFR-файл c помошью jcmd для OpenJDK-приложения с именем Launcher , в котором глубина стека (важно для профилирования spring и других приложений с большим стеком) достаточно длинная и у профайлера задана глубина в 1024 строки

#!/bin/bash -x

# Найти приложение
PID=$(jps | grep Launcher | awk '{print $1}')

# Остановить JFR-сессию с именем profiling, если такая есть
jcmd $PID JFR.stop name=profiling

# Удалить файл от предыдущей сессии profiling
# Если он был создан при предыдущей коменде (JFR.stop)
rm -rf /tmp/app.dump.jfr
rm -rf /tmp/app.dump.jfr.gz

# Поднять глубину стека до 1024
jcmd $PID JFR.configure dumppath=/tmp/app.dump.jfr stackdepth=1024

# Запустить профилирование, используя файл настроек settings.jfc
jcmd $PID JFR.start maxsize=1000M settings=/tmp/settings.jfc filename=/tmp/app.dump.jfr name=profiling dumponexit=true disk=true

# Это просто для отладки - вывести, что сейчас есть 1 активная сессия
jcmd $PID JFR.check
