Не, там вот как:

Cpu 75% - это не как стакан воды, из которого три четверти выпили, но четверть осталась полностью доступна.

Cpu либо занят (полностью) либо свободен (тоже полностью)

А 75% - это значит, что из замера длительностью, например, в 5 секунд, 75% времени cpu был занят и 25% времени свободен

Так вот, когда ты кидаешь новую задачу - проц ставит её в очередь. Для любой задачи выделяются маленькие тайм-слотики и процессор по-очереди занимается каждой задачей кусочек времени.

Но если проц 75% времени уже чем-то занимается, то очевидно, что тебе придётся ждать дольше.

И вот такая модель применяется к каждому самостоятельному ядру отдельно.

Соответственно, раскидав задачи по разным ядрам, ты их рассуёшь по разным очередям и сделаешь чтобы тестируемая система не мешала работе джиметра (но у них обоих будет доступно меньше ресурсов cpu)

1 Go to the bin folder of JMeter apache-jmeter-<version> -> bin
2 Right-click and open jmeter.bat file in notepad or notepad++
3 Search for the keyword ‘set Heap’
4 You can see the default heap size (1 GB – may vary in your JMeter version) set HEAP=-Xms1g -Xmx1g -XX:MaxMetaspaceSize=256m
5 Change the -Xms (initial heap size) and -Xmx (maximum heap size) values. You can increase these values up to 80% of total heap. In my case, I just increased the max heap size value. Refer to the below arguments: set HEAP=-Xms1g -Xmx2g -XX:MaxMetaspaceSize=256m
6 To check the total allocated heap to JVM, run the below command in the command prompt: java -XX:+PrintFlagsFinal -version | findstr /i “HeapSize PermSize ThreadStackSize”
