между shared-iterations и rps + iterations есть разница в результатах

shared-iterations - это load test, max perf, rumpup, spike, ... кто как называет с поиском максимального RPS но с ограничением количества итераций

rps + iterations - это stability, constant-arrival-rate, ... в нем ограничен rps сверху и iterations ограничены

в зависимости от цели применять удобно разное

Да, потоки JMeter это потоки JVM, да они ограничены количеством ресурсов. Как вы видите ваши потоки ждут, они не делают вычислений. Они ждут ответа сервера. В таком случае критическим ресурсом не является CPU, до определенного масштаба, до 1000-2000 потоков

- память, Xmx
- количество сетевых подключений, количество возможных файловых дескрипторов на один процесс и на одного пользователя операционной системы и на всю систему

Плагин работы с потоком может иметь три части

1. Property с пулом подключений (клиентов)
Которые просто открыты и не закрываются

2. Основной sampler (JSR 223) который берет клиента из массива клиентов и делает отправку потока сообщений. Он работает в основной катушке. Его цель отправлять с нужной интенсивностью

🔸 Внести вклад в OpenSource
Желательно создать репозиторий на github.com или gitlab.com и оформить его минимально.
Как инженер по нагрузке может показать свой опыт.

Issue, дефекты, пожелания
Можно оформить issues к существующим OpenSource продуктам. И разместить ссылки на эти issues в Readme.md файле. Продуктами могут быть: Apache.JMeter, Gatling, k6, плагины к ним, Grafana, доски к ней, InfluxDB, Prometheus, VictoriaMetrics, Allure, ...

PR, правки
Можно оформить issue, сделать clone-репозитория, ветку, правку issue, и предложить PoolRequest с исправлением. И тоже зафиксировать информацию об этом в Readme-файле.
Наиболее простые правки могут касаться документации и инструкций.
Правки посложнее могут касаться написания unit-тестов к каким-то частям инструмента
Правки еще посложнее могут касаться параметризации работы инструмента, вот был запрос, и вот в нем появился параметр.
Также можно попробовать известный дефект в инструменте

Без PR, плагины, доски, gist-примеры кода
У многих инструментов есть плагинная система. И можно разработать свой плагин. Это будет отдельный репозиторий, поэтому не потребуется вносить PR во внешний проект. Но сам репозиторий будет иметь связь с известным OpenSource инструментом.
Можно разработать доску мониторинга для #Grafana и опубликовать ее на сайте Grafana. А также репозиторий, где эта разработка ведется и к котому пользователи могут прикреплять issue.

Новый скрипт или инструмент
Проектом может быть новый скрипт. Даже не очень относящийся к нагрузке. Но если его оформить качественно, так что видно - человек постарался. Структура кода понятна. Есть Readme есть конфиг. То этого ценный вклад.
Даже если проект выглядит небрежно, в первой версии, то тоже неплохо - лучше чем ничего

https://docs.influxdata.com/influxdb/v2.6/reference/api/influxdb-1x/dbrp/
https://docs.influxdata.com/influxdb/v2.6/reference/cli/influx/v1/auth/create/
https://docs.influxdata.com/influxdb/v2.6/reference/cli/influx/v1/dbrp/create/

Задать в пемеменных окружения все нужные параметры
set INFLUX_HOST=http://localhost:8086
set INFLUX_ORG=имя-организации (взять из UI influxdb 2)
set INFLUX_TOKEN=токен админа, взять из UI influxdb 2

influx v1 auth create --read-bucket UID_для_Gatling_DB --username gatlingRead
+ задать пароль, пусть gatlingReadPassword

influx v1 dbrp create --bucket-id UID_для_Gatling_DB --db gatling --rp auto --default

А потом подключать уже в Grafana datasource как для InfluxQL

адрес - адрес influxdb v2
база данных - gatling
логин - gatlingRead
пароль - gatlingReadPassword
