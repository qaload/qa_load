Если наш тест длится 1 час, и мы хотим выполнять 10 запусков в сек (первый нижний уровень профиля), то все просто. Нам надо будет выполнить
3600 * 10 = 36 000 итераций
за 3600 сек.
Если в Thread Group сделать
- Number of threads (users) = 36000
- Rump-Up period (seconds) = 3600
- Loop Count = 1

[✓] Delay thread Creation Until needed

То мы получим стабильные 10 запусков в сек в течение одного часа.

09:58
Оптимизация! Обратите внимание на вариант, где Loop Count был равен 10, а не 1. В этом варианте один поток создаётся чтобы выполнить 10 сценариев в цикле. Этот вариант в 10 раз экономичнее чем вариант с Loop Count = 1. Так как ресурсов тратится значительно меньше. Его минус такой. При Loop Count = 1 мы точно знаем, что расписание запуска будет стабильным. Каждую сек один поток создаётся, делает одну итерацию и закрывается. RPS будет более-менее ровный. При Loop Count = 10 поток будет выполнять 10 итераций. По условию задачи, таймеры мы не используем. Потому нагрузка будет менее ровной, но средний RPS на большом периоде времени (10 минут) будет такой же гладкий.

Также обратите внимание на
[✓] Delay thread Creation Until needed
Это позволяет создавать потоки только в нужный момент. К моменту их запуска, предыдущие с высокой вероятностью уже отработают и закроются. И общее количество активных потоков будет невысоким в каждый момент времени.

Теперь задача со звёздочкой. Как сделать вторую ступеньку используя только Thread Group без таймеров.

Пусть нам надо через 20 минут после начала теста поднять нагрузку ещё на 10 сценариев в сек. И продержать такую ступень 15 минут.
И так, нам нужна пауза в 20*60=1200 сек перед началом. За 15 минут мы выполним 15(минут)*60(сек в минуте)*10(сценариев в сек)=9000 итераций. Старты продлятся в течение 15(минут)*60(сек в минуте)=900 сек.

Для такого профиля нам понадобятся такие настройки:

- Number of threads (users) = 9000
- Rump-Up period (seconds) = 900
- Loop Count = 1
[✓] Same User on each iteration
[✓] Delay thread Creation Until needed
[✓] Specify Thread Lifetime
Duration (seconds): не заполнять
Startup delay (seconds): 1200

И если хотим оптимизировать (уменьшить количество стартов потоков):
- Number of threads (users) = 900
- Rump-Up period (seconds) = 900
- Loop Count = 10
[✓] Same User on each iteration
[✓] Delay thread Creation Until needed
[✓] Specify Thread Lifetime
Duration (seconds): не заполнять
Startup delay (seconds): 1200

И второй минус такой реализации открытой модели нагрузки в JMeter. Если сервер зависнет. И ответы перестанут приходить. То потоки будут долго завершать сценарии. И общее количество потоков станет очень большим. Памяти на нагрузочной станции может не хватить.

Чтобы как-то компенсировать зависание можно выставить Duration Assertion на ответы. Или на транзакцию. И в настройках Thread Group настроить - остановку итерации или потока при ошибке.

Duration Assertion на Transaction Controller будет работать только если выставить галочку Generate Parent Sampler. Это не самое оптимальное действие. При такой настройке JMeter будет копить в памяти статистику по запросам внутри транзакции, до ее завершения. И при большом количестве запросов памяти может не хватить. Но при небольших нагрузках - рабочий вариант.

Тогда при срабатывании Duration Assertion произойдет ошибка, Thread Group выполнит действие указанное в настройках: Action to be taken after a Sample Error, и итерация остановится, нагрузка спадёт. Потоки не накопятся. JMeter не упадёт.

простой пример:
твоя бизнес операция включает в себя условное прохождение трех компонент системы, две из которых работают хорошо и быстро, а третья отрабатывает 1 минуту, что не вяжется с SLA. измеряя только бизнес операции, ты делаешь вывод о том, что вся транзакция работает плохо, но если измерять атомарными операциями, условно рпсами, то будет сделан вывод о том, что работает плохо именно тот компонент - а это гораздо более полезно с точки зрения перфоманса, нежели плохая работа всей транзакции.
частенько банкинг сильно неэффективно расходует ресурсы именно по этой причине. простое решение проблемы мерить и тпсы и рпсы, что легко реализуемо в том же jmeter. у тебя есть информация и со стороны бизнес-метрик и со стороны конкретного функционала

Response time - When discussing the response time of the system, people often refers to it as latency. Response time is instead the amount of time a system takes to react to a request once it has received one. For istance, the time between the instant we invoke an API and the instant this API returns the result of its computation, that’s the response time.
10:15
Latency - Latency is sometimes - but not correctly - referred to as the “response time” of a given remote API (e.g. a webservice). When invoking a remote API, the total response time is the sum of the response time proper (the time needed by the system to compute a result) and the time your request (and their response) takes to travel over the network between client and server. The latter is the latency proper.

If we consider the above definitions, when a backend system is being mocked to run performance tests in isolation, the response time of that particular backend is set to zero. If we artificially modify the mock code so that it takes a time > 0 to return the mocked result, it’s a delay we’re adding and not - as it is sometimes referred to - a latency.

Elapsed time. JMeter measures the elapsed time from just before sending the request to just after the last response has been received. JMeter does not include the time needed to render the response, nor does JMeter process any client code, for example Javascript.

Latency. JMeter measures the latency from just before sending the request to just after the first response has been received. Thus the time includes all the processing needed to assemble the request as well as assembling the first part of the response, which in general will be longer than one byte. Protocol analysers (such as Wireshark) measure the time when bytes are actually sent/received over the interface. The JMeter time should be closer to that which is experienced by a browser or other application client.
